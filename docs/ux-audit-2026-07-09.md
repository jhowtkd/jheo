## TL;DR — por que a ferramenta está morrendo

Eu cliquei como você, sem código na mão. Em **todos os fluxos principais** o JHEO
para em uma das três barreiras abaixo:

1. **A stack nem sobe** — Docker Desktop travado, API na 8080 sem ninguém ouvindo,
   Postgres/Redis indisponíveis. Você só vê "API fora do ar" lá em cima. Sem
   ação de recovery.
2. **A UI não diz o que aconteceu quando algo falha** — você clica "Criar
   projeto" e recebe "Unexpected end of JSON input". Você abre um projeto e
   recebe "Falha ao carregar o projeto.". Não dá pra saber se foi o Docker,
   o banco, a rede ou o input.
3. **Quando funciona, o número não confia** — o score é `100 - penalty`
   linear sem ground truth: 1 erro já tira 7 pontos, 15 erros zera a
   categoria, e o "overall" é média simples das categorias que tiveram
   findings (as que não rodaram simplesmente somem). Você não tem como
   defender esse número pra um cliente.

A ferramenta tem a **arquitetura** certa (audit + generate + publish em
monorepo TS, F1-F7 entregues, 26 plugins de auditoria, Flow Producer pra
paralelismo, Findings com diff NEW/UNCHANGED/IMPROVEMENT/REGRESSION). O
problema é a **superfície**: o que aparece pra quem não é dev.

Este relatório lista **24 problemas** divididos em 4 trilhas
(Stack · UX · Veracidade · Polish) e um **plano sem rush** de 4 sprints.

## Como eu testei

Abri `http://127.0.0.1:5173` no Chrome via Playwright e cliquei em tudo:

- `/projects` — lista + criar projeto
- `/templates` — modelos
- `/fixes` — sugestões
- `/settings` — chaves de API
- `/projects/:id` — drill-down do projeto
- `/projects/:id/compose` — composição de geração

Console: **11+ erros 500 por carga de página**, todos
`Failed to load resource: 500 (Internal Server Error) @ /api/health` ou
`/api/projects`. O proxy Vite está mandando pra 8080 e ninguém responde.
Confirmei com `lsof -iTCP:8080 -sTCP:LISTEN` → vazio. `docker info` no

Mesmo sem API, **dá pra mapear a UX lendo o TSX** — empty states, mensagens
de erro, cálculo de score, fluxo de mutações. Isso é metade do relatório.
A outra metade é o que vi na tela.

## Trilha A — Stack (P0: impede usar)

### A1. Docker Desktop em mau estado — CLI trava, daemon não responde
**Sintoma:** `docker info` excede timeout. `docker compose up` retorna
`Error response from daemon: Docker Desktop is unable to start`.
Processos `com.docker.backend services` (PID 67341) e `Docker Desktop`
(PID 67376) estão rodando há 17 minutos consumindo CPU.
**Impacto:** sem Postgres, sem Redis, sem API, sem worker. Tudo abaixo
de 8080 morre.
**Por que o usuário não vê isso:** a UI mostra "API fora do ar" mas não
explica que o Docker caiu, nem dá botão pra "Reiniciar stack", nem mostra
o estado de Postgres/Redis/Worker individualmente.

### A2. Vite proxy devolve 500 quando backend está down — confunde o usuário
**Sintoma:** console mostra `Failed to load resource: 500` em todas as
chamadas `/api/*`. O `vite.config.ts` configura
`proxy: { '/api': 'http://127.0.0.1:8080' }`. Quando o destino não existe,
o `http-proxy-middleware` devolve **500 com body vazio**, daí o
`"Unexpected end of JSON input"` na UI.
**Impacto:** a UI apresenta erro técnico genérico em vez de "Backend
indisponível — tentando reconectar em Xs".

### A3. Sem `docker/.env` versionado — primeiro boot exige Prisma manual
**Sintoma:** `README.md:48-54` diz: "If `POST /api/projects` returns a
Prisma `P2021` (`table does not exist`), push the schema once". Ou seja,
o primeiro boot precisa rodar `docker exec ... npx prisma db push` na mão.
**Impacto:** usuário novo perde 30 min descobrindo isso.

### A4. API health check polling pesado
**Sintoma:** `ProjectDashboard.tsx:50-55` faz `refetchInterval: 5_000`
em health/pages quando há audit rodando. Sem audit, fica parado. Mas
quando volta, dispara 11 fetches por carga.
**Impacto:** em rede ruim, gasta bateria e cria 11 erros por carregamento.

## Trilha B — UX (P1: ferramenta confunde)

### B1. Página Modelos é uma caixa vazia
`/templates` mostra: título, subtítulo, **dois skeletons** e nada mais.
Sem botão "Criar modelo". Sem empty state. Sem explicação.
Reproduzido: screenshot `02-templates.png`.
**Causa:** `TemplatesList.tsx` carrega templates via React Query; quando
a query falha (sem API), o estado `isLoading` vira `true` eterno (ou
permanece após erro), e o componente só renderiza skeletons nesse
estado. Não tem branch de erro nem empty state com CTA.

### B2. Settings mostra só skeletons quando vazio
`/settings` exibe "Chaves salvas" + dois skeletons. Mesmo problema de B1:
loading eterno em vez de empty state com CTA "Adicionar chave".
Reproduzido: screenshot `04-settings.png`.

### B3. Empty state de projetos sem ilustração
`/projects` (vazio): o componente `EmptyState` existe no código mas
só aparece quando `projects.data.length === 0 && !projects.isLoading`.
Sem API: `projects.isLoading` fica `true` eterno → esqueleto vazio.
E mesmo se aparecesse: o CTA "Criar projeto" só foca o primeiro input,
não cria nada novo.
Reproduzido: screenshot `01-projects-list.png`.

### B4. "Falha ao carregar o projeto." sem contexto
`ProjectDashboard.tsx:134` — `project.isError` renderiza literalmente
essa string. Sem botão "Tentar de novo", sem dica, sem causa.
Reproduzido: `06-dashboard-empty.png` ao abrir
`/projects/cmrc0p08a001bktp35js3hyct`.

### B5. "Unexpected end of JSON input" exposto ao usuário
Quando o Vite proxy devolve 500 com body vazio, o `api.ts` faz
`r.json()` que joga `SyntaxError: Unexpected end of JSON input`. A UI
captura como `create.error.message` e mostra a string técnica literal.
Reproduzido: `05-after-create.png` ao clicar "Criar projeto".

### B6. Criar projeto: input diz "URL", backend recebe `domain`
`ProjectsList.tsx:99` placeholder `"https://acme.com.br"`.
`api.ts:52` aceita `{ domain: string }`. Sem validação no client de
"é uma URL válida" vs "é um domínio".
**Impacto:** usuário digita `https://acme.com.br/promocoes`, manda,
servidor recorta, e o audit roda no lugar errado (ou dá 404 de sitemap).

### B7. "Rodar auditoria" sem configuração alguma
`/projects/:id/audit` é uma página com **um único botão "Iniciar
auditoria"**. Sem:
- limite de páginas (`maxPages` é 0 = sem cap, configurado via POST body)
- filtros (sempre roda tudo: sitemap + crawl + root)
- quais plugins vão rodar (todos, sempre)
- preview de quantas páginas serão descobertas
- estimativa de tempo

**Impacto:** dispara um job de 30+ minutos sem aviso, em site grande,
sem volta.

### B8. Botão "Gerar" desabilita sem explicar por quê
`GenerationComposer.tsx:52`:
```ts
submitDisabled = !prompt || (!templateId && !activeTemplate) || create.isPending;
```
Sem tooltip, sem hint textual. Se não tem template ativo, o botão fica
cinza e o usuário não sabe que precisa criar um template antes.
**Impacto:** "clico em Gerar e não cria nada" — na verdade nem habilita.

### B9. Sem feedback em tempo real durante audit
`ProjectDashboard.tsx:173-226` mostra progresso de páginas, mas:
- só aparece se há `lastAudit`
- sem ETA
- sem cancel funcional testável ("Cancelar auditoria" é fácil de clicar,
  mas se a API caiu, também não dá feedback)

### B10. Findings sem "como corrigir"
`FindingList` mostra categoria, severity, mensagem. Sem link pra
documentação do plugin, sem exemplo de fix, sem botão "Sugerir correção"
na própria lista (só existe `/fixes` que abre outro fluxo).

### B11. Empty state de Fixes parece erro
`/fixes` (sem projeto): "Você ainda não tem projetos." + botão "Ir para
projetos". Isso **não é erro**, é vazio. Mas o visual é o mesmo de uma
página quebrada (espaço vazio, texto pequeno, sem ilustração).
Reproduzido: `03-fixes.png`.

## Trilha C — Veracidade (P2: não dá pra confiar no número)

### C1. Cálculo de score é heurístico sem ground truth
`packages/core/src/audit/score.ts` (42 linhas, todo o score engine):
```ts
const WEIGHTS = { error: 7, warning: 3, info: 1 } as const;
const score = Math.max(0, 100 - entry.penalty);
const overall = populatedCount === 0 ? 100 : Math.round(populatedTotal / populatedCount);
```

Problemas:
- **1 erro = 93, 15 erros = 0** (cap linear sem saturação gradual)
- **info pesa 1**, então 100 infos zeram a categoria — não faz sentido,
  info deveria ser neutro ou peso baixo
- **`null` é ignorado no overall** — categoria que não rodou (sem
  findings porque plugin falhou) simplesmente some da média. Um site
  com 0 findings em `cwv` porque o Lighthouse crashou vai ter overall
  alto falsamente
- **não normaliza por página** — 5 findings em 1 página = 5 findings
  em 500 páginas no score
- **sem peso por categoria** — SEO com 5 erros = CWV com 5 erros
  (na vida real SEO pesa mais pra muitos negócios)

**Por que você não confia:** com razão. O número é bonito mas arbitrário.

### C2. Sem comparação histórica visível
Tem `auditHistory` no dashboard mas o score mostrado é o do último
audit. Não tem "antes/depois", "tendência nas últimas N auditorias",
"score melhorou X% desde mês passado". O usuário não sabe se está
indo bem ou pior.

### C3. Findings sem evidência
A UI mostra a `message` do finding mas não exibe o `evidence` que o
plugin coletou (linha 88 do `api.ts` diz que tem
`evidence: Record<string, unknown>` — mas `FindingList` provavelmente
não renderiza). Sem ver o que foi medido, o usuário não consegue
validar.

### C4. CWV é categoria mas o AuditRunner não avisa que precisa de browser
Para o plugin de CWV rodar (Lighthouse / Puppeteer), o worker precisa
de Chromium instalado. O `Dockerfile.api` instala, mas se o usuário
roda API standalone (`pnpm --filter @jheo/api run dev`), não tem.
Sem aviso de "CWV vai falhar neste modo".

### C5. Página de resultado do audit mostra ScoreCard sem explicar pesos
`AuditResults.tsx:60-78` passa um objeto com `pagesTotal: 0,
pagesWithError: 0, lastAuditAt: null` hardcoded pro ScoreCard. Ou
seja, o `pagesAudited` aparece mas `pagesTotal` e `pagesWithError`
sempre zerados. ScoreCard não reclama, mostra barra de progresso
enganosa.

## Trilha D — Polish (P3: faz parecer ferramenta de verdade)

### D1. Sidebar com só 4 itens esconde fluxos importantes
Audit, Materials, Channels, Generations, Publishes ficam em
drill-downs a 2 cliques. Para alguém que abre a ferramenta pela
primeira vez, a sidebar diz "JHEO tem 4 coisas". Não dá pra ver
o ciclo de vida.

### D2. Categorias do score sem tradução
`ScoreCard.tsx:8`:
```ts
const CATEGORIES = ['seo', 'cwv', 'geo', 'a11y', 'content'] as const;
```
Renderiza literal em minúscula (`cat` no `textTransform: 'uppercase'`).
Usuário vê "SEO", "CWV", "GEO", "A11Y", "CONTENT". Sem tooltip,
sem tradução. CWV = Core Web Vitals — só dev sabe.

### D3. "ESPAÇO DE TRABALHO" como label da sidebar
Trade-mark design choice, mas quebra pra quem chega pela primeira
vez. Esperaria "Projeto atual", "Workspace", "Menu" ou simplesmente
nada.

### D4. Header "GEO · SEO · CONTEÚDO" sem contexto
3 letras-palavras sem explicar o que cada uma faz. Tem 26 plugins
de auditoria mas o usuário não sabe o que "GEO" cobre.

### D5. i18n keys em inglês nas URLs
`/projects`, `/templates`, `/fixes`, `/settings`. Se um dia
precisar vender pra outro mercado, vai ter que reescrever rotas
ou conviver com inconsistência.

### D6. Footer "Usuário local · uso único · v0.1.0"
"Uso único" é confuso — single-user mode? Trial? Free tier? Sem
explicação.

### D7. Sem dark/light toggle
Forçado em dark mode. Em ambientes claros, vai estourar retina.

### D8. Erros da API sem traduções amigáveis
`create.error.message` é literalmente a mensagem do Error. Sem
mapa tipo `apiErrorToHuman(error) → "Sem conexão com o servidor"`.

## Plano sem rush — 4 sprints de 1 semana

Cada sprint termina com **uma coisa que o usuário consegue usar end-to-end**.
Sem dependência entre sprints (pode paralelizar).

### Sprint 1 — "A ferramenta liga" (3-5 dias) → destrava TUDO
**Objetivo:** usuário clica em "Subir stack" e tudo funciona sem tutorial.

- [ ] **A1.** Diagnosticar Docker Desktop travado. Script de bootstrap:
      `bin/dev-up.sh` que (a) verifica `docker info` em loop com timeout,
      (b) se falhar, sugere reiniciar Docker Desktop via `osascript`,
      (c) se sucesso, roda `docker compose up -d --build`, (d) espera
      `/api/health` retornar 200, (e) roda `prisma db push` se necessário.
      **Sem isso, nada abaixo importa.**
- [ ] **A2.** Vite proxy: configurar `selfHandleResponse: true` e
      responder 503 com `{ error: 'backend_unavailable' }` em vez de 500
      vazio. UI: mostrar `Retry-After` e reconectar.
- [ ] **A3.** Versionar `docker/.env.example` com todas as vars;
      tornar `prisma db push` idempotente no Dockerfile.
- [ ] **B4+B5.** Substituir mensagens de erro em todos os `*.tsx` por
      helper `humanError(err)` que mapeia: `network error → "Sem conexão
      com o servidor"`, `Unexpected end of JSON → "O servidor caiu —
      tente novamente"`, etc. Sem string técnica vazada.

**Critério de aceite:** usuário roda `bin/dev-up.sh`, abre
`http://127.0.0.1:5173`, vê "API conectada" no topo, consegue criar
projeto e rodar audit básico em < 2 minutos.

### Sprint 2 — "Consigo usar sem perguntar" (1 semana)
**Objetivo:** cada botão da ferramenta tem propósito óbvio, fluxo de
happy path leva do zero ao "publiquei" sem tutorial.

- [ ] **B1+B2+B3.** Empty states com ilustração + CTA em todas as listas
      (Projects, Templates, Materials, Channels, Fixes, Settings).
      Branch de erro com botão "Tentar de novo" + dica ("Backend fora?
      Rode `pnpm run dev-up`").
- [ ] **B6.** Validar URL no client (regex simples), alinhar placeholder
      e label com o que o backend espera (`domain` ou `rootUrl` —
      padronizar).
- [ ] **B7.** `AuditRunner` ganha formulário antes do botão: `maxPages`
      (default 50), `incluir (sitemap/crawl/root)`, `estimativa de
      tempo` (rough: `pages × 8s + overhead`). Botão "Pré-visualizar
      URLs" antes de "Iniciar".
- [ ] **B8.** `GenerationComposer`: hint visível "Você precisa de um
      template ativo em /templates antes de gerar" quando botão está
      disabled.
- [ ] **B9.** Progress de audit com ETA (rolling avg de páginas/segundo).
- [ ] **D1.** Sidebar com 7 itens (Projetos, Auditorias, Modelos,
      Materiais, Gerações, Correções, Canais). Auditorias globais em
      `/audits` mostrando todas as rodadas recentes.
- [ ] **D8.** Mapeamento de erros da API → mensagens humanas.

**Critério de aceite:** usuário novo abre a ferramenta, segue o fluxo
visual sem ler documentação, consegue ir do "criar projeto" ao
"publiquei em WordPress" sozinho.

### Sprint 3 — "O número é defensável" (1 semana)
**Objetivo:** o score tem explicação que sobrevive a pergunta de cliente.

- [ ] **C1.** Reescrever `score.ts` com:
      - Normalização por página (`penalty / pageCount`)
      - Pesos por categoria (SEO: 0.25, CWV: 0.20, GEO: 0.25,
        A11Y: 0.15, Content: 0.15 — ajustável)
      - Categorias sem findings = `null` exibido como "—" com tooltip
        "Plugin não rodou" (em vez de sumir da média)
      - `info` não penaliza score (só conta achados)
      - Curva não-linear: erros acumulam com peso quadrático
        (`penalty += error^1.2`) pra evitar "1 erro zera tudo"
- [ ] **C2.** `ScoreCard` ganha badge "vs último audit" (↑ 5 / ↓ 3 / =
      0) + sparkline das últimas 5 auditorias.
- [ ] **C3.** `FindingList` expande findings com `evidence` colapsável,
      link "Por que isso importa?" pra doc do plugin.
- [ ] **C5.** `AuditResults` consertar ScoreCard: passar `pagesTotal` e
      `pagesWithError` reais, não hardcoded.
- [ ] **D2.** Traduzir categorias no ScoreCard (i18n keys), com tooltip
      explicando cada sigla.

**Critério de aceite:** usuário consegue printar o dashboard e levar
pra reunião com cliente sem precisar explicar o que é CWV ou por que
o número é 87.

### Sprint 4 — "Parece ferramenta, não protótipo" (1 semana)
**Objetivo:** polish que faz a diferença na percepção.

- [ ] **B10.** Em cada finding, botão "Sugerir correção" inline
      (delega ao fluxo de /fixes existente).
- [ ] **B11.** Empty state de Fixes com ilustração + texto "Quando
      você rodar auditorias, sugestões de correção aparecerão aqui".
- [ ] **D3+D4.** Sidebar com label claro + tooltip explicando o que
      cada categoria cobre.
- [ ] **D5.** URLs internacionalizadas (`/projetos`, `/modelos`, etc).
- [ ] **D6.** Footer explica o "uso único" ("Esta instância roda só
      pra você — sem login, sem sync na nuvem").
- [ ] **D7.** Dark/light toggle persistido em localStorage.
- [ ] **C4.** `AuditRunner` avisa "CWV precisa de browser — instale
      Chromium se rodando fora do Docker".
- [ ] Telemetria básica: contagem de clicks, tempo em página,
      taxa de erro por endpoint (sem PII).

**Critério de aceite:** alguém abre a ferramenta pela primeira vez
e diz "ok, isso é um produto" em vez de "isso é um protótipo".

## Observação sobre o estado atual

**A ferramenta tem o esqueleto certo.** Você entregou F1 (audit) →
F7 (suggestions) em ~2 meses de planejamento (vejo no
`.planning/STATE.md` e no README). A infra tá em pé, os 26 plugins
existem, o Flow Producer escalou pra 500 páginas no cenbrap.edu.br.
O incidente de 2026-07-08 (QueueEvents pub/sub) mostra que você está
caçando bugs reais no nível certo.

**O gap é UX, não engenharia.** E UX é onde mora a decisão de compra
do cliente. O relatório acima ataca em 4 sprints sem mexer na
arquitetura — só na superfície e no motor de score.

Sem rush, priorize o **Sprint 1**. Se você só fizer aquilo, você
consegue abrir a ferramenta de manhã sem 20 minutos de ritual.
